# Git 커밋 메시지 규칙 및 브랜치(main, dev) 사용법 안내

---

## 1. main과 dev 브랜치는 이렇게 써요!

- **main 브랜치**
  - 항상 정상적으로 동작하는 코드만!
  - 실제로 제출하거나, 다른 사람과 공유하는 공식 버전 역할
  - 웬만하면 실험적인 코드를 바로 넣지 않아요

- **dev 브랜치**
  - 개발, 실험, 수정… 마음껏 작업하면서 쓰는 브랜치!
  - 여기서 여러 번 커밋하고 다듬은 다음 main에 반영(merge)

### 브랜치 사용 흐름 예시
1. dev에서 자유롭게 개발/수정하고 커밋
2. 동작이 잘 되는지 확인
3. main 브랜치로 병합(merge)  
   → main은 항상 작동 보장! dev는 자유롭게 수정

## 2. 커밋 메시지 규칙

- "한 줄 요약" + "필요하면 상세 설명"  
- 아래 형식 지키면 한눈에 보기 좋아요!
```
[타입]: 한 줄 요약 (최대 50자)

상세 설명 (선택)
```

#### 예시
```
feat: 카메라 이미지 저장 기능 추가

이미지 데이터를 jpg로 저장하는 함수를 새로 만들었어요.
main.py에 save_image() 함수 추가!
```

---

## 3. 커밋 타입 분류

| 타입     | 의미                                   |
|----------|----------------------------------------|
| feat     | 새로운 기능 추가                       |
| fix      | 버그(오류) 수정                        |
| docs     | 문서(README 등) 내용 수정              |
| style    | 코드 스타일/들여쓰기/공백 등만 수정    |
| refactor | 코드 구조만 개선 (기능 변화 없음)      |
| test     | 테스트(실험) 코드 추가/수정            |
| chore    | 빌드, 환경설정 등 기타 작업            |

---

## 4. 커밋 메시지 쓸 때 꿀팁

- "무엇을, 왜, 어떻게" 바꿨는지 담백하게!
- 동사로 시작(예: 추가, 수정, 삭제 ...)
- 너무 길게 한 번에 적지 말고, 적당히 쪼개서 자주 커밋하자
- 한글/영어 둘 다 사용 가능. 자유롭게!

---

## 5. 커밋 메시지 실전 예시

```
feat: 라이다 센서 데이터 파싱 기능 추가

fix: 카메라 연결 오류 수정

docs: README 사용법 예시 추가

style: 코드 들여쓰기 통일

refactor: 이미지 처리 함수 구조 개선

test: 카메라 캘리브레이션 테스트 코드 추가

chore: .gitignore에 cache 폴더 추가
```
